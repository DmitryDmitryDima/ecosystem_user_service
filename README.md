# User Service
User service на данный момент находится в зародышевом состоянии — ему будет отведена ответственность за все социальные взаимодействия вне пользовательских комнат, а также хранение дополнительных user properties. В планах решить, как будет организовано хранение таких данных, как файлы в диалогах, аватарки, блоги и т д — скорее всего это будет выделенное cloud хранилище, а сам сервис в properties будет хранить лишь ссылку на них. Возможно объединение планируемого user storage с этим функционалом, однако жизнеспособность этой идее будет зависеть от того, каким образом будет работать rag в storage сервисе.

Используемые технологии - Spring Web, Spring Data JPA (PostgreSQL), Spring AMQP (RabbitMQ),  Spring Validation.


В User Service, как и в других сервисах, вся информация ассоциирована по user uuid – данные о username и role не дублируются. Исходя из этого, многие запросы к этому сервису предполагают автоматическую конвертацию targetUsername в targetUUID (помещаемый в security context наряду с viewer uuid, username и role) на этапе прохождения через связку api gateway + auth service.


На данный момент единственной таблицей в бд является UserProperties, содержащая поля:
- avatarLink;
- userUUID;
- about

Создание сущности пользовательских данных происходит лениво — при первом соответствующем запросе (страничка профиля). Раньше это происходило в результате ивента, прилетающего с auth сервиса, но от этой схемы было решено отказаться.

User service – один из главных кандидатов на внедрение выделенного из java projects service outbox фреймворка. Пример использования — загрузка в чат большого файла (сущность файла будет ассоциирована с чатом и, соответственно, доступна только его участникам):
- Загрузка файла во временное хранилище — создание сущности файла со статусом processing с назначенным uuid. Запуск outbox цепочки.
- В рамках цепочки — загрузка файла в cloud
- В рамках цепочки — изменение статуса в db на uploaded
- Ивент для внешней системы
